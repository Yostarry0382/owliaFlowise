## 1. 開発初学者向け（ストア利用）

開発経験が浅い、またはAIエージェント構築が初めてのユーザー向けのユースケースです。

### 1.1 勉強会資料の自動生成

**ユーザー**: 開発チームのリーダー

**課題**: 毎月の技術勉強会でパワーポイント資料を作るのに半日かかる

**エージェント構成**:
```
[RAG: 社内Wiki] + [RAG: 技術ブログ] + [RAG: 過去の勉強会資料]
        ↓
[プロンプト: 勉強会構成テンプレート]
        ↓
[LLM: 内容生成]
        ↓
[人間確認: 内容レビュー]
        ↓
[出力: PowerPoint形式]
```

**使い方**:
1. 「今月のテーマ: Kubernetes入門」と入力
2. 3つのRAGから関連情報を自動収集
3. 過去資料のトンマナを踏襲した構成案を生成
4. 人間確認で微調整 → PowerPoint出力

**得られる効果**: 半日 → 30分に短縮。毎回同じ品質の資料

---

### 1.2 議事録 → タスク抽出 → チケット起票

**ユーザー**: プロジェクトマネージャー

**課題**: 会議後の議事録整理、タスク抽出、Backlog起票が面倒

**マルチエージェント構成**:
```
[議事録整理エージェント] → [タスク抽出エージェント] → [チケット起票エージェント]
```

**使い方**:
1. 会議の文字起こしテキストを入力
2. 議事録エージェント: 発言を整理・要約
3. タスク抽出エージェント: 「誰が・何を・いつまで」を抽出
4. 起票エージェント: Backlog APIでチケット自動作成

**得られる効果**: 会議後30分の作業 → 3分で完了

---

### 1.3 社内規程FAQ + エスカレーション判定

**ユーザー**: バックオフィス担当

**エージェント構成**:

```
        ↓

        ↓
[条件分岐: 判断が必要な場合]
```

2. RAGから規程を検索 → 回答生成
3. グレーゾーンの質問は自動で「担当部署へ」と案内

**得られる効果**: 問い合わせ対応の70%を自動化

## 2. 開発者向け（PG・PM・PMO）

### 2.1 プログラマー（PG）向け

#### 2.1.1 既存コードの仕様理解

**ユーザー**: 中途入社の開発者

**課題**: 引き継いだレガシーコードの仕様がわからない。ドキュメントも古い

**エージェント構成**:
```
[RAG: ソースコード] + [RAG: 設計書] + [RAG: 過去のPR/コミット履歴]
        ↓
[LLM: コード解説生成]
        ↓
[出力: 処理フロー図 + 日本語説明]
```

**使い方**:
1. 「OrderService.javaの処理フローを説明して」と入力
2. コード・設計書・変更履歴を横断検索
3. 「この関数は○○を行い、△△テーブルを更新します」と解説

**得られる効果**:
- コードリーディング時間を60%短縮
- 先輩への質問回数を削減

---

#### 2.1.2 エラーログ解析 → 原因特定

**ユーザー**: バックエンド開発者

**課題**: 本番エラーの原因調査に毎回時間がかかる

**エージェント構成**:
```
[入力: エラーログ]
        ↓
[RAG: 過去の障害対応履歴] + [RAG: ソースコード]
        ↓
[LLM: 原因推定 + 修正箇所の提案]
        ↓
[人間確認: 対応方針決定]
```

**使い方**:
1. スタックトレースを貼り付け
2. 過去の類似エラーと対応履歴を検索
3. 「原因: NullPointerException @ UserService:142。過去に同様のケースがあり、○○で解決」

**得られる効果**:
- 原因特定時間を50%短縮
- 過去の知見を即座に活用

---

#### 2.1.3 テストコード自動生成

**使い方**:

2. プロジェクトの既存テストパターンを参照して生成
3. 正常系・異常系・境界値のテストケースを提案

**得られる効果**:
- テスト作成時間を70%短縮
- カバレッジ向上

#### 2.1.4 API仕様書の自動生成・更新

**ユーザー**: バックエンド開発者

**課題**: コードとAPI仕様書が乖離している。手動更新が面倒

```
[LLM: 仕様書生成]
        ↓
[差分検出: 既存仕様書との比較]
        ↓
[出力: 更新されたAPI仕様書（Markdown/Swagger）]
```

**CI/CD連携**:
```yaml

- name: Update API Docs
  run: curl -X POST /api/flows/execute -d '{"flowId": "api-doc-gen"}'
```

**得られる効果**:
- 仕様書の自動更新
- コードと仕様書の乖離を防止

## 付録: OwliaFabricaの主要機能

### エージェント作成に使用できるノード

| ノード種別 | 用途 |
|-----------|------|
| LLMノード | GPT、Claude、Geminiなどの言語モデル処理 |
| RAGノード | 社内ドキュメント検索 |
| プロンプトテンプレート | 再利用可能なプロンプト定義 |
| メモリノード | 会話履歴の管理 |
| ツールノード | 外部API連携（Backlog、Slack等） |
| 条件分岐ノード | 条件による処理分岐 |
| 人間確認ノード | Human-in-the-Loopによる承認 |
| 出力ノード | ファイル出力（Excel、PowerPoint、PDF等） |

### マルチエージェント機能

複数のOwlAgentを連携させて、より複雑なワークフローを構築可能：

- **順次実行**: エージェントA → エージェントB → エージェントC
- **並列実行**: 複数エージェントを同時実行して結果を統合
- **条件分岐**: 結果に応じて次のエージェントを選択

### 外部連携

- **Slack/Teams**: Bot経由でエージェントを呼び出し
- **CI/CD**: GitHub Actions、Jenkins等からAPI実行
- **業務システム**: REST APIによる連携
- **定期実行**: スケジューラーによる自動実行
